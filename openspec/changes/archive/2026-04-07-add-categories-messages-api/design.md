## Context

The `api-rs` subproject is an empty Gradle stub. This change bootstraps it as a Spring Boot 4+ application providing two REST endpoints (categories and messages) secured with OAuth2. GitHub acts as the identity provider via an OAuth App named "emails-summarizer". The `api-rs` service is a pure OAuth2 Resource Server — it validates GitHub access tokens on every request. It uses an embedded H2 database so the service runs standalone aside from the GitHub dependency.

## Goals / Non-Goals

**Goals:**
- Scaffold `api-rs` as a runnable Spring Boot 4+ application.
- `GET /categories` — returns all categories (name, code, description).
- `GET /messages?categoryCode={code}` — returns messages (id, title, body) for a given category.
- OAuth2 Resource Server protecting both endpoints — validates GitHub Bearer tokens.
- Custom `OpaqueTokenIntrospector` calling `https://api.github.com/user` to validate tokens.
- H2 in-memory database with schema and sample data loaded at startup.
- Update `api-rs/build.gradle` with Spring Boot Gradle plugin and all required dependencies.

**Non-Goals:**
- Write/mutate endpoints (POST, PUT, DELETE).
- Embedded authorization server (Spring Authorization Server removed in favour of GitHub).
- Client Credentials grant type — GitHub OAuth Apps do not support it.
- Production database.
- Frontend integration.
- Pagination or sorting on list endpoints.

## Decisions

### Decision: GitHub as OAuth2 identity provider (Authorization Code only)

**Choice**: Register a GitHub OAuth App named "emails-summarizer". The `ui` performs the Authorization Code flow against GitHub, obtains an opaque GitHub access token, and sends it as a Bearer token to `api-rs`.

**Rationale**: GitHub is an externally managed, well-known identity provider requiring no infrastructure to run. Developers already have GitHub accounts. This eliminates the need for an embedded authorization server and any local user management.

**Constraint**: GitHub OAuth Apps only support the Authorization Code grant type. Client Credentials (machine-to-machine) is not available — this capability is removed from scope.

**Alternative considered**: Embedded Spring Authorization Server — self-contained but requires managing users, keys, and token lifecycle; replaced by GitHub.

---

### Decision: api-rs as OAuth2 Resource Server with custom opaque token introspection

**Choice**: Configure Spring Security as an OAuth2 Resource Server (`spring-boot-starter-oauth2-resource-server`) with a custom `OpaqueTokenIntrospector` that validates a GitHub access token by calling `GET https://api.github.com/user` with `Authorization: Bearer <token>`. A successful response (HTTP 200) means the token is valid; the GitHub login name is used as the principal name.

**Rationale**: GitHub tokens are opaque (not JWTs), so standard JWKS-based JWT validation cannot be used. Calling the GitHub user info endpoint is the canonical way to validate a GitHub OAuth token. Spring Security's `OpaqueTokenIntrospector` interface provides a clean extension point for this.

**Alternative considered**: Spring Security OAuth2 Login (session-based) — suitable for server-rendered apps but not for a stateless REST API consumed by a SPA.

---

### Decision: GitHub client ID and secret stored in `application.yml`

**Choice**: Store `spring.security.oauth2.resourceserver.opaquetoken.client-id` and `client-secret` (the GitHub OAuth App credentials) in `application.yml`, with placeholders for local dev overridden via environment variables in any deployment.

**Rationale**: Simplest configuration approach for local development. In production, inject secrets via environment variables or a secrets manager — never commit real values to source control.

---

### Decision: H2 in-memory database with schema.sql + data.sql

**Choice**: `spring.datasource.url=jdbc:h2:mem:emailsdb`, schema defined in `schema.sql`, sample rows in `data.sql`.

**Rationale**: Zero-configuration persistence that resets cleanly on each restart. Spring Boot auto-runs these scripts at startup when H2 is on the classpath with `spring.jpa.hibernate.ddl-auto=none`.

---

### Decision: Message ID as UUID

**Choice**: `id` column is `UUID` type, generated by the database (H2 `RANDOM_UUID()`).

**Rationale**: Matches the requirement specification and avoids leaking sequential integer keys in the API response.

---

### Decision: Package structure under `com.emailssummarizer.apirs`

**Choice**:
```
com.emailssummarizer.apirs
├── ApiRsApplication.java
├── category/
│   ├── Category.java
│   ├── CategoryRepository.java
│   └── CategoryController.java
├── message/
│   ├── Message.java
│   ├── MessageRepository.java
│   └── MessageController.java
└── security/
    ├── GitHubOpaqueTokenIntrospector.java
    └── ResourceServerConfig.java
```

---

### Decision: `GET /messages?categoryCode={code}` query parameter style

**Choice**: `categoryCode` as a query parameter.

**Rationale**: Matches the requirement. Simpler to implement with `@RequestParam` + repository derived query.

## Risks / Trade-offs

- **GitHub API dependency at runtime** → Every API request to `api-rs` triggers a call to `https://api.github.com/user`. If GitHub is down or rate-limited, token validation fails. Mitigation: add short-lived in-memory caching of validated tokens in a follow-up.
- **GitHub rate limiting** → Unauthenticated introspection calls count towards rate limits. The GitHub OAuth App credentials sent with each introspection call raise the limit significantly; ensure they are always configured.
- **Opaque tokens cannot be validated offline** → Unlike JWTs, GitHub tokens require a network call to validate. Acceptable for this phase.
- **GitHub OAuth App client secret in config** → Must not be committed to source control with real values. Use environment variable overrides (`GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET`) in any non-local environment.
- **H2 data resets on restart** → Expected and acceptable for this phase.
- **H2 console exposed** → `spring.h2.console.enabled=true` MUST be disabled in any deployed environment.
- **CORS** → The `ui` origin must be allowed on `api-rs` for browser requests to succeed; configure `@CrossOrigin` or a global `CorsConfiguration`.
